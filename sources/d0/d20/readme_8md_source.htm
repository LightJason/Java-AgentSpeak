5 Based on the project [Jason](http://jason.sourceforge.net/) by Jomi F. HÃ¼bner and Rafael H. Bordini a Java 9 implementation has been build-up with parallel execution calls. The version defines an additional [AgentSpeak(L)](https://en.wikipedia.org/wiki/AgentSpeak) grammar based on [AntLR](http://www.antlr.org/) for simulating a [multi-agent system](https://en.wikipedia.org/wiki/Multi-agent_system) with a fuzzy-based logical calculus and grammar features like lambda expressions. Agent execution based on a mathematical structure to describe an optimizing process by a [finite-state-machine](https://en.wikipedia.org/wiki/Finite-state_machine)

24 ## Base Definitions
25 
26 ### Belief
27 
28 * Beliefs implicitly describe the current state of the agent
29 * Beliefs will be updated before the cycle is run (beliefbase uses an update mechanism)
30 * Beliefs must be exists iif an expression is computed (beliefs can be exist on the fly)
31 * Belief addition triggers a plan with the definition ```+belief```
32 * Belief retraction triggers a plan with the definition ```-belief```
33 * Belief modification with ```-+``` does not exists anymore
34 * [Variables](#variable) within a belief literal will be unified before the belief is added to the beliefbase
35 
36 
37 ### Action
38 
39 * Actions will be run immediately
40 * Actions can fail (fuzzy-logic false) or succeed (fuzzy-logic true)
41 * There is no difference between internal and external actions
42 * Actions with ```@```-prefix wil be executed in parallel (each inner action will be run in parallel)
43 
44 
45 ### Plan
46 
47 * Plans are _sequences of [actions](#action), [rules](#rule) and/or achievement / test [goals](#goal)_
48 * Plans has got an optional context, that defines a constraint for execution (default is fuzzy-logic true and matches always)
49 * Plans fail iif the defuzzyfication returns fuzzy-logic false
50 * Plans returns a boolean value which defines fail (fuzzy-logic false) and success (fuzzy-logic true)
51 * Plans run items in sequential order on default
52 * If the plan calls an _achievement [goal](#goal) addition_, the [goal](#goal) will be added for the next cycle
53 * An _achievement [goal](#goal) deletion_ does not exists anymore
54 
55 #### Internals Constants
56 
57 * The plan has got additional [constant variables](#variable), that are added in the context condition (values are calculated before plan execution is started)
58  * _PlanFail_ stores the number of fail runs and _PlanFailRatio_ normalized value in [0,1]
59  * _PlanSuccessful_ stores the number of successful runs and _PlanSuccessfulRatio_ normalized value in [0,1]
60  * _PlanRuns_ number of runs of the plan (fail + successful runs)
61 
62 #### Fuzziness
63 
64 * Fuzzy value must be in [0,1]
65 * Each [action](#action) in a fuzzy-plan returns also a fuzzy value to define the fuzziness
66 * The [plan](#plan) or [rule](#rule) result returns fuzzy-logic true / false and the aggregated fuzzy value
67 
68 
69 ### Rule
70 
71 * Rules are similar to [plans](#plan) without the context condition
72 * Rules cannot be triggered by a goal, so they must be called from a plan
73 * Rules will be executed with the prefix ```$```
74 * Rules run immediatly
75 * Rules run sequentially on default
76 * Rules returns a fuzzy-logic result for success or fail
77 * [Variables](#variable) will be passed, so if a rules succeed the value of the variable will be passed back to the calling plan
78 
79 
80 ### Rule / Plan Annotation
81 
82 * Annotations can modify a plan / rule behaviour to change runtime semantic
83 * The following annotation can be used
84  * ```@Constant( AnyValue, 5 )``` creates the given constant variable
85  * ```@Atomic``` the plan / rule cannot be fail, it returns always true (only the [actions](#action) can fail)
86  * ```@Parallel``` all items will be run in parallel
87 
88 ### Goal
89 
90 * Semantically a goal marks a certain state of the world an agent _wishes to bring about_ [AgentSpeak, p.40]
91 * _Achievement goals_ triggers an _achievement goal addition_ which leads to the execution of a corresponding [plan](#plan)
92 * On agent start, there can exists one _initial goal_ only (like the ```main``` function in Java or C/C++)
93 * Each agent can track _more than one goal_ at the same time otherwise the agent idles (the suspending state is not used)
94 * Goals are triggered by external events which will match by the goal name
95 * Goals will be resolved into [plans](#plan) with equal name (and allowed context), the [plan](#plan) is the intantiiation of the goal
96 * Goals are run in parallel independed from other goals
97 * A goal is a sequence of [plans](#plan) which must all finished completly
98 * A goal is part of exactly one [intention](#intention)
99 * If a goal can match a [desire](#desire) (the goal is near to the desire) it can add an event to match the desire [belief](#belief)
100 * If the agent is in sleeping / hibernate state and the ```wakeup``` method is called, it triggers the wakeup-goal
101 
102 #### Test Goals
103 
104 * A test goal is an atom with the definition ```?literal```
105 * The test return true iif a plan with an equal literal is within the current execution context (current running)
106 
107 
108 ### Intention
109 
110 * An intention is the _convex hull_ of its [goals](#goal)
111 * The intention is set of of goals, which must exist simultaneously
112 * Intentions cannot be in conflict with other intentions, so there dies not exists any overlaping
113 
114 
115 ### Desire
116 
117 * A Desire is a vertex of the edge of all intentions
118 * Desires are defined by a set of beliefs
119 * Desires can be in conflict with other desires, represented that the desires have got a large distance (much as possible)
120 * The desire is successfully reached, iif all beliefs are existing anytime

122 
123 ### Variable
124 
125 * Variables are written with an upper-case letter at begin
126 * Thread-safe variables for parallel runtime start with ```@``` (at-sign) followed by an upper-case letter
127 * Variables can store a literal or string to call a [rule](#rule) or [plan](#plan) e.g. ```!X(3,2)``` calls a plan or ```$X(2,1)``` calls a rule
128 
129 ### Action / Term Annotation
130 
131 * In LightJason one can specify HOW actions and terms will be executed / unified.
132 * Concept of ```action-term-annotation```s allows to annotate ```actions```, and ```terms``` to perform
133  * unification (```>>```)
134  * parallel execution (```@```), see [Variables](#variable) and _lambda expressions_.
135  * ...
136 * If more than one ```action-term-annotation``` is needs to be added, they have to be ordered according to the rule: _First HOW, then WHAT_, e.g. ```@>>``` (parallel unification)
137 * To annotate multiple actions/terms brackets ```(```,```)``` can be used. See the following examples
138 * Examples
139  * ```@>>( foo(X), X > 1 ) && Value > 0.5``` (unify ```foo(X)``` and ```X > 1``` in parallel and if this results in a true statement check whether ```Value > 0.5```)
140  * ```>>foo(X) && X > 1 && Value > 0.5``` (unify ```foo(X)```, then test the following terms sequentially)
